# IGK-TodoList 도메인 정의서 품질 평가 보고서

## 평가일: 2025-11-25

## 평가 대상: IGK-TodoList 도메인 정의서 v1.0

---

## 1. 완전성 (Completeness)

**점수: 8/10**

### 강점

- **핵심 엔티티 완벽 식별**: User, TodoItem, Trash 3개의 핵심 엔티티가 명확하게 정의됨
- **비즈니스 규칙 포괄적 정의**: 인증/권한, 할일 생명주기, 제약사항이 체계적으로 구조화됨
- **유스케이스 완전성**: 사용자 관리, 할일 관리, 휴지통 관리의 3개 영역으로 모든 기능 영역 커버
- **도메인 용어집 제공**: 핵심 용어 6개에 대한 명확한 정의 포함
- **할일 유형 구분**: 개인 할일과 공통 할일의 구분이 명확히 정의됨

### 개선점

- **할일 완료 상태 누락**: 요구사항에는 명시되지 않았으나, 할일 관리 앱의 핵심 기능인 "할일 완료" 개념이 정의되지 않음
- **국경일 데이터 관리 방식 미정의**: 공통 할일(국경일)이 어떻게 시스템에 등록되고 관리되는지에 대한 설명 부족
- **시작일시와 만료일시의 관계 미정의**: 두 일시 간의 선후 관계, 필수/선택 여부가 명확하지 않음
- **에러 케이스 미정의**: 잘못된 입력, 권한 없는 접근 등 예외 상황에 대한 정의 부족

### 권장사항

1. 할일 상태에 "완료(Completed)" 상태 추가 및 완료 처리 비즈니스 규칙 정의
2. 공통 할일의 등록/관리 프로세스를 별도 섹션(4.4 공통 할일 관리)으로 추가
3. 엔티티 속성에 필수/선택 여부 및 제약조건 명시 (예: 시작일시 <= 만료일시)
4. 섹션 7로 "예외 상황 및 에러 처리" 추가

## 2. 명확성 (Clarity)

**점수: 7/10**

### 강점

- **명확한 구조화**: 개요 → 엔티티 → 개념 → 규칙 → 유스케이스 → 용어집의 논리적 흐름
- **용어 일관성**: "휴지통", "원복/복원", "영구삭제" 등 핵심 용어가 문서 전체에서 일관되게 사용됨
- **할일 유형의 명확한 구분**: 개인 할일과 공통 할일의 차이가 명확히 정의됨
- **생명주기 명확성**: 활성 → 휴지통 → 영구삭제의 단계가 잘 설명됨

### 개선점

- **"상태" 속성의 모호성**: TodoItem의 "상태" 속성이 있으나, 섹션 3.2의 할일 상태(활성/휴지통/영구삭제)와의 관계가 불분명
- **"만료일시"의 의미 모호성**: 용어집에서 "목표 완료 시간"으로 정의했으나, 만료 시 자동 처리가 있는지 불명확
- **Trash 엔티티와 상태의 중복**: Trash를 별도 엔티티로 정의하면서 동시에 TodoItem의 상태로도 관리하는 것처럼 보여 혼란 야기
- **"소유자" 속성 모호성**: TodoItem의 "소유자" 속성이 User를 참조하는지, 공통 할일의 경우 NULL인지 불명확

### 권장사항

1. 할일 상태 관리 방식 명확화: "Trash 엔티티는 삭제된 할일의 메타데이터를 저장하며, TodoItem의 상태는 활성/삭제됨으로만 구분" 등으로 명시
2. 각 엔티티 속성에 데이터 타입, NULL 허용 여부, 참조 관계 추가
3. "만료일시" 정의 개선: "할일의 마감 기한으로, 시스템은 만료일시가 지나도 자동 처리하지 않으며 사용자 참고용" 등으로 구체화
4. 공통 할일의 소유자 처리 방식 명확화 (NULL 또는 시스템 계정)

## 3. 추적성 (Traceability)

**점수: 9/10**

### 강점

- **요구사항 완벽 매핑**: 원본 요구사항의 모든 항목이 도메인 정의서에 추적 가능
  - "사용자마다 할일 목록" → User 엔티티 + 소유자 속성
  - "만료 기한" → 시작일시, 만료일시 속성
  - "국경일 관리" → 공통 할일 개념
  - "로그인 사용자만" → 인증 및 권한 규칙
  - "휴지통 및 원복" → Trash 엔티티 + 복원 유스케이스
- **기능-규칙 연결**: 각 유스케이스가 비즈니스 규칙과 명확히 연결됨
- **제약사항 추적**: "공통 할일 삭제 불가" 요구사항이 제약사항 4.3에 명시됨

### 개선점

- **추적성 메트릭스 부재**: 요구사항 ID와 도메인 요소 간 매핑 테이블이 없어 양방향 추적이 어려움
- **우선순위 정보 누락**: 어떤 요구사항이 필수(Must)이고 선택(Should)인지 구분 없음

### 권장사항

1. 섹션 8로 "요구사항 추적 매트릭스" 추가:
   ```
   | 요구사항 ID | 요구사항 내용 | 도메인 요소 | 우선순위 |
   |------------|--------------|------------|---------|
   | REQ-001    | 사용자별 할일 관리 | User, TodoItem.소유자 | Must |
   ```
2. 각 유스케이스에 관련 요구사항 ID 태깅

## 4. 검증 가능성 (Verifiability)

**점수: 6/10**

### 강점

- **명확한 권한 규칙**: "로그인한 사용자는 자신의 할일만" 등 테스트 가능한 조건
- **상태 전이 명확성**: 활성 → 휴지통 → 영구삭제의 단계별 전이 조건 명시
- **불가능한 작업 명시**: "공통 할일은 수정 및 삭제 불가" 등 부정 케이스 정의

### 개선점

- **검증 기준 부재**: 비즈니스 규칙에 대한 구체적 검증 조건이 없음
  - 예: "자신의 할일만"을 어떻게 검증할 것인가? (소유자ID == 로그인사용자ID)
- **경계값 미정의**: 제목/내용의 최대 길이, 날짜 범위 등 제약 조건 부재
- **테스트 시나리오 부재**: 각 유스케이스에 대한 예상 입력/출력 예시 없음
- **성공/실패 조건 불명확**: 예를 들어 "할일 수정"이 성공했다는 것을 어떻게 확인하나?

### 권장사항

1. 각 비즈니스 규칙에 검증 기준 추가:
   ```
   4.1.2 자신의 할일만 접근 가능
   - 검증 조건: TodoItem.소유자 = 현재_로그인_사용자.ID
   - 위반 시: 권한 없음 오류 반환
   ```
2. 엔티티 속성에 제약 조건 추가:
   ```
   - 제목: 문자열, 1-200자, 필수
   - 내용: 문자열, 0-2000자, 선택
   - 시작일시: DateTime, 선택, 만료일시보다 이전이어야 함
   ```
3. 각 유스케이스에 "성공 조건"과 "실패 조건" 섹션 추가
4. 부록으로 "테스트 시나리오 예시" 추가

## 5. 유지보수성 (Maintainability)

**점수: 7/10**

### 강점

- **모듈화된 구조**: 섹션이 명확히 분리되어 특정 부분만 수정 가능
- **확장 가능한 설계**: 할일 유형, 할일 상태 등이 열거형으로 정의되어 향후 추가 용이
- **버전 관리 가능성**: 문서 구조가 Git 등 버전 관리 시스템과 호환
- **용어집 제공**: 새로운 팀원의 온보딩에 유용

### 개선점

- **변경 이력 부재**: 문서의 수정 이력을 추적할 방법이 없음
- **버전 정보 누락**: 문서 버전, 작성자, 승인자 등 메타데이터 부재
- **변경 영향도 분석 어려움**: 한 요소를 변경했을 때 영향받는 다른 요소들이 명시되지 않음
- **참조 문서 부재**: 관련 기술 문서, API 명세, DB 스키마 등 다른 문서와의 연결 고리 없음

### 권장사항

1. 문서 상단에 메타데이터 섹션 추가:
   ```markdown
   ## 문서 정보

   - 버전: 1.0
   - 최종 수정일: 2025-11-25
   - 작성자: [이름]
   - 검토자: [이름]
   - 승인자: [이름]
   - 상태: Draft / Review / Approved
   ```
2. 변경 이력 섹션 추가:
   ```markdown
   ## 변경 이력

   | 버전 | 일자       | 변경 내용 | 작성자 |
   | ---- | ---------- | --------- | ------ |
   | 1.0  | 2025-11-25 | 초안 작성 | [이름] |
   ```
3. 각 엔티티/규칙에 의존 관계 명시:
   ```
   ### 2.2 TodoItem
   - 참조: User (소유자)
   - 참조됨: Trash
   - 관련 규칙: 4.1.2, 4.2.1, 4.3.1
   ```
4. 섹션 9로 "관련 문서" 추가하여 API 명세, DB 스키마, 테스트 계획 등 연결

### 예시 1: 엔티티 속성 상세화

```markdown
### 2.2 TodoItem (할일)

#### 속성 정의

| 속성명   | 타입     | 필수 | 제약조건          | 기본값 | 설명                    |
| -------- | -------- | ---- | ----------------- | ------ | ----------------------- |
| 할일ID   | UUID     | O    | PK, Unique        | Auto   | 할일 고유 식별자        |
| 제목     | String   | O    | 1-200자           | -      | 할일 제목               |
| 내용     | Text     | X    | 0-2000자          | NULL   | 할일 상세 설명          |
| 시작일시 | DateTime | X    | 만료일시보다 이전 | NULL   | 할일 시작 시간          |
| 만료일시 | DateTime | X    | 시작일시보다 이후 | NULL   | 할일 마감 시간          |
| 상태     | Enum     | O    | 활성/삭제됨       | 활성   | 할일 현재 상태          |
| 소유자   | UUID     | X    | FK(User)          | NULL   | 소유 사용자 (NULL=공통) |
| 유형     | Enum     | O    | 개인/공통         | 개인   | 할일 유형               |
| 생성일시 | DateTime | O    | -                 | Now()  | 생성 시간               |
| 수정일시 | DateTime | O    | -                 | Now()  | 최종 수정 시간          |
```

### 예시 2: 검증 가능한 비즈니스 규칙

```markdown
### 4.1.2 자신의 할일만 접근 가능

**규칙**: 로그인한 사용자는 자신이 소유한 개인 할일만 조회/수정/삭제할 수 있다.

**검증 조건**:

- 개인 할일: `TodoItem.소유자 = 현재_로그인_사용자.ID`
- 공통 할일: 모든 사용자 조회 가능

**성공 케이스**:

- 입력: 사용자A가 자신의 할일ID=123 조회 요청
- 조건: TodoItem(123).소유자 = 사용자A.ID
- 출력: 할일 정보 반환

**실패 케이스**:

- 입력: 사용자A가 사용자B의 할일ID=456 수정 요청
- 조건: TodoItem(456).소유자 = 사용자B.ID ≠ 사용자A.ID
- 출력: 403 Forbidden 오류, "본인의 할일만 수정할 수 있습니다"

**테스트 방법**:

1. 사용자A로 로그인
2. 사용자B의 할일ID로 수정 API 호출
3. 403 오류 응답 확인
```
